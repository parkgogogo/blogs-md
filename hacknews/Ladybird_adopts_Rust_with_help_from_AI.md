[Ladybird adopts Rust, with help from AI](https://ladybird.org/posts/adopting-rust/) ([via](https://news.ycombinator.com/item?id=47120899)) Really interesting case-study from Andreas Kling on advanced, sophisticated use of coding agents for ambitious coding projects with critical code. After a few years hoping Swift's platform support outside of the Apple ecosystem would mature they switched tracks to Rust their memory-safe language of choice, starting with an AI-assisted port of a critical library:

[Ladybird 在 AI 的帮助下采用 Rust](https://ladybird.org/posts/adopting-rust/)（[来源](https://news.ycombinator.com/item?id=47120899)）这是一篇来自 Andreas Kling 的非常有趣的案例研究，讲述了如何在涉及关键代码的雄心勃勃的编程项目中，高级而 sophisticated 地使用编程代理。在几年间一直希望 Swift 在苹果生态系统之外的平台支持能够成熟之后，他们转而选择了 Rust 作为他们首选的内存安全语言，从一个由 AI 辅助移植的关键库开始：

Our first target was LibJS , Ladybird's JavaScript engine. The lexer, parser, AST, and bytecode generator are relatively self-contained and have extensive test coverage through [test262](https://github.com/tc39/test262), which made them a natural starting point.

我们的第一个目标是 LibJS，Ladybird 的 JavaScript 引擎。词法分析器、解析器、AST 和字节码生成器相对自成体系，并且通过 [test262](https://github.com/tc39/test262) 拥有广泛的测试覆盖，这使它们成为自然的起点。

I used [Claude Code](https://docs.anthropic.com/en/docs/claude-code) and [Codex](https://openai.com/codex/) for the translation. This was human-directed, not autonomous code generation. I decided what to port, in what order, and what the Rust code should look like. It was hundreds of small prompts, steering the agents where things needed to go. [...]

我使用 [Claude Code](https://docs.anthropic.com/en/docs/claude-code) 和 [Codex](https://openai.com/codex/) 进行翻译。这是由人指导的，而非自主代码生成。我决定要移植什么、按什么顺序移植，以及 Rust 代码应该是什么样子。这是数百个小提示，引导代理前往需要去的地方。[...]

The requirement from the start was byte-for-byte identical output from both pipelines. The result was about 25,000 lines of Rust, and the entire port took about two weeks. The same work would have taken me multiple months to do by hand. We've verified that every AST produced by the Rust parser is identical to the C++ one, and all bytecode generated by the Rust compiler is identical to the C++ compiler's output. Zero regressions across the board.

从一开始的要求就是两个管道输出字节级完全相同的结果。最终产生了约 25,000 行 Rust 代码，整个移植过程耗时约两周。同样的工作如果我手工完成可能需要几个月。我们已经验证了 Rust 解析器产生的每一个 AST 都与 C++ 版本完全相同，Rust 编译器生成的所有字节码也与 C++ 编译器的输出完全相同。完全没有回归问题。

Having an existing conformance testing suite of the quality of test262 is a huge unlock for projects of this magnitude, and the ability to compare output with an existing可信实现 makes agentic engineering much more of a safe bet.

拥有像 test262 这样质量的现有合规测试套件，对于如此规模的项目来说是一个巨大的优势，而且能够与现有可信实现进行输出比较的能力，使得代理工程变得更加安全可靠。

## 批判性思考评论

这篇文章展示了一个令人瞩目的里程碑：AI 编程代理正在从"玩具"转变为可用于关键基础设施生产的工具。以下是几点值得深入思考的观察：

**1. 测试套件的决定性作用**
文章特别强调了 test262 合规测试套件的重要性。这揭示了一个关键洞察：AI 辅助开发的最大瓶颈不是代码生成能力，而是验证 correctness 的能力。当存在一个权威的测试基准时，AI 的潜力才能真正释放。这提示我们，投资于测试基础设施可能比单纯追求更好的模型更有价值。

**2. "人机协作"模式的成熟**
Andreas 强调这是"human-directed, not autonomous"——人指导而非自主完成。这种描述勾勒出了未来编程的图景：人类仍然是架构师和决策者，而 AI 成为高效的执行者。数百个小提示而非一个巨大的提示，这种迭代式的、渐进的方法可能是使用 AI 进行复杂工程任务的最优模式。

**3. 效率提升的量化**
两周 vs 几个月——这种数量级的效率提升是颠覆性的。但值得警惕的是，这种效率提升高度依赖于特定条件：明确的验证标准（test262）、模块化架构（self-contained components）、以及经验丰富的工程师指导。这不是"用 AI 取代程序员"，而是"让优秀程序员成为超级程序员"。

**4. Rust 的战略选择**
从 Swift 转向 Rust 的决策反映了浏览器引擎这类基础软件的现实考量：跨平台支持和内存安全是不可妥协的需求。AI 加速了这一迁移，但技术选型的根本原因仍然是人做出的判断。

**5. 字节级相同的输出要求**
这种极端严格的要求实际上创造了一个完美的 AI 应用场景：目标是明确可量化的（字节级相同），验证是自动化的（测试套件），人类只需要解决例外情况和边界条件。

总体而言，这个案例研究为 AI 辅助软件工程提供了一个实用的、可复制的模式，而不是过度承诺的炒作。它表明，AI 的最大价值可能不在于替代人类，而在于放大人类的能力——特别是那些已经建立了良好工程实践的人类团队。
