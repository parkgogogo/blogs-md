# 研究报告：SWE-CI - 评估 Agent 在持续集成循环中维护代码库的能力

**日期**: 2026-03-18
**论文链接**: [https://arxiv.org/abs/2603.03823](https://arxiv.org/abs/2603.03823)
**GitHub**: [https://github.com/SKYLENAGE-AI/SWE-CI](https://github.com/SKYLENAGE-AI/SWE-CI)

---

## 1. 核心概述
**SWE-CI** 是第一个基于**持续集成（CI）循环**的存储库级基准测试，旨在将代码生成的评估范式从短期的“功能正确性”转向动态、长期的“**可维护性**”。

以往的基准（如 SWE-bench）多关注“一次性修复”（Snapshot-style），而 SWE-CI 要求 Agent 在平均 **233 天**、**71 次连续提交**的历史跨度中，通过多轮迭代解决复杂的需求变更。

## 2. 关键创新
### 2.1 评估范式的转变
*   **Snapshot-based (旧)**: 接收单一需求，产生一次性补丁。
*   **Evolution-based (SWE-CI)**: 需求是动态产生的。Agent 的前期决策会影响后续迭代的难度。

### 2.2 EvoScore (进化得分)
引入了一个名为 **EvoScore** 的代理指标。它采用未来加权均值（future-weighted mean），给予后期迭代更高的权重。
*   如果 Agent 为了快速通过测试而积累了“技术债”，导致后续修改困难，其 EvoScore 会随之下降。
*   只有具备良好架构设计、能产出可扩展代码的 Agent 才能在长期进化中获得高分。

### 2.3 架构师-程序员双 Agent 协议
模拟真实开发流程：
*   **Architect Agent**: 识别功能差距，编写高层级自然语言需求文档（每次不超过 5 个最紧急需求）。
*   **Programmer Agent**: 根据需求文档执行代码修改。

## 3. 数据集构建 (SWE-CI Dataset)
*   **来源**: 从 4,923 个高质量 Python 存储库中筛选。
*   **规模**: 100 个任务，来自 68 个不同存储库。
*   **难度**: 每个任务涉及至少 1,000 行代码变更（中位历史跨度 233 天）。
*   **环境**: 每个任务都配套了预构建的 Docker 镜像，确保环境可重现。

## 4. 实验观察与结论
论文评估了来自 8 个供应商的 18 个模型：
1.  **能力加速**: 2026 年之后发布的模型在代码维护能力上有显著飞跃。
2.  **供应商差异**:
    *   **Claude Opus 系列**: 在整个观察期内处于绝对领先地位。
    *   **GLM-5**: 表现也非常强劲。
    *   **策略偏向**: MiniMax、DeepSeek 和 GPT 更注重长期收益；而 Kimi 和 GLM 偏向短期回报。
3.  **回归控制挑战**: 大多数模型的“零回归率”（Zero-regression rate）低于 0.25。只有 Claude Opus 系列能超过 0.5。这意味着即使是顶尖模型，在长期维护中也难以完全避免破坏已有功能。

## 5. 师兄参考意义
SWE-CI 的出现提醒我们，评估 AI 程序员不能只看它能不能修好当前的 Bug，更要看它修 Bug 的方式是否优雅、是否会给未来埋坑。在构建自己的 Agent 系统时，引入“架构师+程序员”的协同模式，并关注长期代码质量，是走向成熟 AI 软件工程的关键。
