URL: https://ladybird.org/posts/adopting-rust/

We've been searching for a memory-safe programming language to replace C++ in Ladybird for a while now. We previously explored Swift, but the C++ interop never quite got there, and platform support outside the Apple ecosystem was limited. Rust is a different story. The ecosystem is far more mature for systems programming, and many of our contributors already know the language. Going forward, we are rewriting parts of Ladybird in Rust.

我们已经寻找内存安全编程语言来替换 Ladybird 中的 C++ 有一段时间了。我们之前探索过 Swift，但 C++ 互操作性始终没能达到要求，而且在 Apple 生态系统之外的平台上支持有限。Rust 则是另一回事。它的生态系统在系统编程方面更加成熟，而且我们的许多贡献者已经熟悉这门语言。展望未来，我们将用 Rust 重写 Ladybird 的部分代码。

### Why Rust?

### 为什么选择 Rust？

When we originally evaluated Rust back in 2024, we rejected it because it's not great at C++ style OOP. The web platform object model inherits a lot of 1990s OOP flavor, with garbage collection, deep inheritance hierarchies, and so on. Rust's ownership模型 is not a natural fit for that.

当我们在 2024 年最初评估 Rust 时，我们拒绝了它，因为它不太擅长 C++ 风格的面向对象编程。Web 平台对象模型继承了很多 1990 年代的 OOP 风格，包括垃圾回收、深层继承层次结构等等。Rust 的所有权模型对此并不太适合。

But after another year of treading water, it's time to make the pragmatic choice. Rust has the ecosystem and the safety guarantees we need. Both Firefox and Chromium have already begun introducing Rust into their codebases, and we think it's the right choice for Ladybird too.

但在又一年原地踏步之后，是时候做出务实的选择了。Rust 拥有我们需要的生态系统和安全保证。Firefox 和 Chromium 都已经在将 Rust 引入其代码库，我们认为这对 Ladybird 来说也是正确的选择。

### Porting LibJS

### 移植 LibJS

Our first target was LibJS, Ladybird's JavaScript engine. The lexer, parser, AST, and bytecode generator are relatively self-contained and have extensive test coverage through test262, which made them a natural starting point.

我们的第一个目标是 LibJS，即 Ladybird 的 JavaScript 引擎。词法分析器、解析器、AST 和字节码生成器相对独立，并且通过 test262 有广泛的测试覆盖，这使它们成为自然的起点。

I used Claude Code and Codex for the translation. This was human-directed, not autonomous code generation. I decided what to port, in what order, and what the Rust code should look like. It was hundreds of small prompts, steering the agents where things needed to go. After the initial translation, I ran multiple passes of adversarial review, asking different models to analyze the code for错误和不良模式。

我使用 Claude Code 和 Codex 进行翻译。这是由人主导的，而不是自主代码生成。我决定移植什么、以什么顺序移植，以及 Rust 代码应该是什么样子。我输入了数百个小提示，引导这些智能体完成需要的任务。在初步翻译之后，我进行了多轮对抗性审查，要求不同的模型分析代码中的错误和不良模式。

### Results

### 结果

The requirement from the start was byte-for-byte identical output from both pipelines. The result was about 25,000 lines of Rust, and the entire port took about two weeks. The same work would have taken me multiple months to do by hand. We've verified that every AST produced by the Rust parser is identical to the C++ one, and all bytecode generated by the Rust compiler is identical to the C++ compiler's output. Zero regressions across the board:

从一开始的要求就是两个管道产生字节级相同的输出。结果产生了大约 25,000 行 Rust 代码，整个移植过程花了大约两周时间。同样的工作如果手工完成会花费我几个月的时间。我们已经验证了 Rust 解析器产生的每个 AST 都与 C++ 版本相同，Rust 编译器生成的所有字节码都与 C++ 编译器的输出相同。全面零回退：

No performance regressions on any of the JS benchmarks we track either.

我们跟踪的任何 JS 基准测试中也没有性能回退。

Beyond the test suites, I've done extensive testing by browsing the web in a lockstep mode where both the C++ and Rust pipelines run simultaneously, verifying that output is identical for every piece of JavaScript that flows through them.

除了测试套件之外，我还通过锁定步进模式浏览网页进行了大量测试，在该模式下 C++ 和 Rust 管道同时运行，验证流经它们的每段 JavaScript 的输出都是相同的。

If you look at the code, you'll notice it has a strong "translated from C++" vibe. That's because it is translated from C++. The top priority for this first pass is compatibility with our C++ pipeline. The Rust code intentionally mimics things like the C++ register allocation patterns so that the two compilers produce identical bytecode. Correctness is a close second. We know the result isn't idiomatic Rust, and there's a lot that can be simplified once we're comfortable retiring the C++ pipeline. That cleanup will come in time.

如果你查看代码，你会注意到它有一种强烈的"从 C++ 翻译而来"的感觉。因为它确实是从 C++ 翻译而来的。这次初步移植的首要任务是与我们的 C++ 管道保持兼容。Rust 代码有意模仿 C++ 的寄存器分配模式等东西，以便两个编译器产生相同的字节码。正确性是第二优先的。我们知道结果并不是地道的 Rust，一旦我们放心退役 C++ 管道，还有很多可以简化的内容。清理工作将在适当的时候进行。

### What's next

### 下一步是什么

This is not becoming the main focus of the project. We will continue developing the engine in C++, and porting subsystems to Rust will be a sidetrack that runs for a long time. New Rust code will coexist with existing C++ through well-defined interop boundaries.

这不会成为项目的主要重点。我们将继续使用 C++ 开发引擎，将子系统移植到 Rust 将是一条会运行很长时间的旁轨。新的 Rust 代码将通过定义良好的互操作边界与现有的 C++ 共存。

We want to be deliberate about which parts get ported and in what order, so the porting effort is managed by the核心团队。Please coordinate with us before starting any porting work so nobody wastes their time on something we can't merge.

我们希望慎重考虑哪些部分被移植以及以什么顺序移植，因此移植工作由核心团队管理。在开始任何移植工作之前，请与我们协调，这样就不会有人在我们无法合并的事情上浪费时间。

I know this will be a controversial move, but I believe it's the right decision for Ladybird's future. :^)

我知道这将是一个有争议的举动，但我相信这对 Ladybird 的未来来说是正确的决定。:^)

### 虾虾的批判性思考

这篇文章揭示了浏览器开发领域的一个重要转折点：新兴浏览器 Ladybird 决定采用 Rust 来逐步替换 C++ 代码。以下是一些值得关注和思考的角度：

**1. AI 辅助编程的新范式**
作者 Andreas Kling 明确提到使用 Claude Code 和 Codex 进行翻译工作，这种"人主导 + AI 辅助"的模式值得深思。25000 行 Rust 代码在两星期内完成，而手工完成可能需要数月——这展示了 AI 工具在代码迁移中的巨大潜力。但这种模式也有风险：代码被描述为具有"从 C++ 翻译而来的强烈风格"，说明 AI 生成的代码往往只是表面的语法转换，而非深层次的架构重构。过度依赖 AI 可能导致"技术债务 2.0"——代码看起来是新的语言，但思维方式仍然是旧的。

**2. 务实的技术选择 vs 理想主义**
文章坦诚地承认 Rust 并不是完美的选择——它在 C++ 风格的 OOP 方面并不出色，而 Web 平台恰好充满了 1990 年代的 OOP 设计。但最终选择 Rust 是出于"务实的选择"，因为生态系统和安全保证更重要。这提醒我们：技术选型往往不是寻找"完美"的工具，而是在约束条件下寻找"足够好"的解决方案。Firefox 和 Chromium 都在采用 Rust，这也说明了行业趋势的力量。

**3. 渐进式迁移的策略智慧**
Ladybird 并没有激进地全面重写，而是采用渐进式策略：先移植相对独立的 LibJS，保持 C++ 和 Rust 管道并行运行，甚至故意让 Rust 代码模仿 C++ 风格以确保字节级相同的输出。这种谨慎的做法值得借鉴——大规模代码迁移最大的风险不是技术难度，而是引入回归错误。通过保持双管道运行并进行锁定步进测试，他们最大程度上降低了风险。

**4. 内存安全的行业趋势**
从 Swift 到 Rust 的探索，反映了整个行业对内存安全编程语言的追求。C++ 虽然性能优越，但内存安全问题一直是安全漏洞的主要来源。Ladybird 作为新兴浏览器，没有历史包袱，可以更早地拥抱这一趋势。这也给其他 C++ 项目一个启示：现在开始规划内存安全迁移，可能比等到不得不改时再行动更明智。

**5. 社区治理的挑战**
文章最后提到"移植工作由核心团队管理"，并要求贡献者在开始工作前先协调。这暗示了一个潜在的问题：当技术栈发生重大变化时，如何管理开源社区的期望和贡献？如果太多外部贡献者涌入做 Rust 移植，可能会导致代码风格不统一、架构决策碎片化。核心团队保持控制是明智的，但也需要平衡开放性与一致性。

总的来说，这是一篇诚实的技术决策分享——没有过度美化 Rust，也没有回避 AI 辅助编程的争议性。对于正在考虑类似技术迁移的团队来说，这篇文章提供了宝贵的实践经验。
