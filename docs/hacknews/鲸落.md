Title: 鲸落
URL: https://nesbitt.io/2026/02/21/whale-fall.html
Body: 当一头鲸鱼在开阔的海洋中死去，它的尸体会沉入深渊海底，成为一个生态系统。海洋生物学家将这种现象称为「鲸落」（whale fall），而尸体会在三个相互重叠的阶段中维持生命：移动清道夫在数月内撕食软组织，富集机会主义者会在数年内占据骨骼及其周围的沉积物，化能合成细菌则会在数十年间以骨骼本身为食，将储存在骨骼中的脂质转化为能量，支持整个特殊生物群落。一次单独的鲸落就能在原本贫瘠的海底维持生命长达五十年。

迈克尔·温瑟（Michael Winser）在我们讨论被遗弃项目的依赖图会发生什么时随口提到了鲸落，而这个词在我脑海中挥之不去。

一个大型开源项目无人维护了。也许维护者精疲力竭，也许背后的公司转型了。项目停止更新，但并未消失。它静静地躺在 GitHub 上积累着 issue，最后一次提交的时间越来越远，然后有人 fork 了它来合并最紧急的补丁。如果项目足够受欢迎，会出现多个 fork，像盲鳗争夺鲸脂一样竞争用户，不过大多数 fork 很快就消亡了，只有一两个凭借某个人有足够的时间或机构支持来维持而存活下来。OpenOffice 就是这样变成了 LibreOffice，MySQL 变成了 MariaDB，Hudson 变成了 Jenkins——每一个都是清道夫式的 fork，通过一个熟悉的流程成长为正统替代品：fork 公告、迁移指南，以及「为什么你应该切换」的博客文章。

随后，更小的项目开始提取特定模块，或构建针对已死项目数据格式的工具。Google Reader 不是开源的，但它关闭时也发生了同样的情况：Feedly、Miniflux、FreshRSS、Tiny Tiny RSS 等十几个项目争先恐后地填补真空，其中好几个实现了 Google Reader API 或 Fever API——不是因为这些 API 有多好，而是因为多年的 RSS 客户端都是按照这些接口构建的。许可证无关紧要。接口是公开的，其他软件依赖于它们，这就足够了。

然后，结构性的骨架——协议、文件格式和 API 契约——继续支撑着特殊的社区，而这些社区甚至可能根本不知道这些骨骼从何而来。OpenDocument 格式比创造它的 OpenOffice 社区活得更久，由数十个语言生态系统中的文档格式库维持着。Docker 在 2015 年将其容器运行时和镜像格式捐赠给了开放容器倡议（Open Container Initiative）。OCI 规范现在定义了容器的工作原理，无论运行时是什么。Docker 自身的统治地位逐渐消退；规范却没有。Tree-sitter 是为 Atom 构建的，在 GitHub 将 Atom 归档后，它成为了 Zed、Neovim、Helix 以及过去几年发布的大多数编辑器内部的语法引擎。

### 演替

我注意到无人维护的库存在一个反复出现的模式：连续性的重新定殖。一个项目沉寂了，有人 fork 它，其他项目开始依赖这个 fork，然后那个 fork 的维护者也精疲力竭，整个循环在小规模上重复。每一代 fork 通常都比前一代更小，贡献者更少，用户群更窄，直到最终这个概念本身迁移了，而不是代码。某个其他语言生态系统中的人看着堆积的残骸，决定从头重写这个概念，带着设计理念前进，但将实现抛在身后。

Sass 经历了这个过程。最初的参考实现是一个 Ruby gem。当 Ruby 的性能成为瓶颈时，LibSass 用 C++ 重写了它，sassc gem 则为 Ruby 用户包装了它。然后 LibSass 本身被弃用，取而代之的是 Dart Sass，现在这是正统实现。这个概念在十年间从 Ruby 迁移到 C++ 再到 Dart，每一次重写都受益于前代积累的 bug 报告和设计争论，而在每个阶段，都有其他生态系统中的包装库以 Sass 语言规范的结构骨架为食。今天大多数写 Sass 的人根本不知道它最初是一个 Ruby gem。

连续性重新定殖有一个糟糕的失败模式。Edera 在 Rust 的 tar-rs 库中发现了一个差异解析 bug，影响了下游的每一个 fork：tar-rs 本身、async-tar、tokio-tar，以及 Astral 等公司维护的多个内部 fork（他们的 fork 内置于 uv 包管理器中）。协调披露意味着要联系一个碎片化的 fork 图谱中约二十个实体，其中四个库版本中有三个无人维护，还有几个维护者无法联系。这个漏洞存在的原因是代码通过连续的 fork 被复制转发，而没有人重新审计它们都从原始版本继承的 PAX 头解析。这个 bug 一直存在于骨骼中，被每一个 fork 继承。发现某个包的哪些 fork 受到安全公告影响是一个我正在单独研究的问题，因为目前没有人有好的工具来解决它。

CentOS 在转向 Stream 之后，是同样模式在操作系统规模上的体现：Rocky Linux 和 AlmaLinux 分出了 fork，更小的 RHEL 兼容重建版本在它们周围的富集层出现，而下层的结构性骨架——RPM 打包、systemd 惯例、文件系统层级——无论哪个特定发行版在任何给定时刻是死是活，都保持不变。

### 许可证变更

当一个项目从开源许可证切换到源代码可用许可证时，清道夫阶段几乎立即触发，往往在变更生效之前就开始了。Redis 到 Valkey，Elasticsearch 到 OpenSearch，Terraform 到 OpenTofu——这个模式现在已经足够熟悉，社区已经将其简化为一种惯例：rush 去 fork 最后一个开放的提交，短暂竞争，然后整合到一两个幸存者周围。这些情况下，生物体并没有完全死亡。Redis 这个产品仍然有收入和路线图。但从开源生态系统的角度来看，代码主体已经停止接受外部贡献，fork 开始偏离原始版本，就像 MariaDB 偏离 MySQL 那样。

抽象层是持续存在的部分。每个集成了开放版本的项目都面临一个选择：是跟随专有版本还是切换到 fork，而很多人干脆构建一个兼容垫层。这些垫层往往比引发它们的争议活得更久，在许可证辩论冷却多年后，仍然以原始 API 的骨架为食。

### Sun Microsystems

甲骨文在 2010 年收购 Sun，与其说是单个鲸落，不如说是一群鲸鱼同时在海上死去。Java、Solaris、ZFS、DTrace、VirtualBox、NetBeans、GlassFish、Hudson、MySQL，每一个都沉到了不同的海底，各自引发了演替。有的产生了单一的统治性 fork（Hudson 到 Jenkins，ZFS 到 OpenZFS），有的分散成相互竞争的谱系（仅 MySQL 就滋养了 MariaDB、Percona，以及短暂的 Drizzle，而 Drizzle 本身在被遗弃后成为了一个更小的鲸落），还有的在基金会之间辗转后才稳定下来（NetBeans 到 Apache，GlassFish 到 Payara 以及更广泛的 Jakarta EE 生态系统）。下层的结构性骨架——JVM 字节码格式、ZFS 磁盘格式、MySQL 线协议——在开发者从未听说过 Sun 的项目中仍然承担着负载。

### 浅水区

有些项目在浅水区死去，尸体会被快速回收。收购雇佣（Acqui-hire）就是这样运作的：公司被吞并，代码变成专有或被归档，知识随着人员分散而不是积累在其他人可以取食的公共尸体中。企业整合也有类似效果，因为当一个大型独立项目被折叠进平台公司的专有服务时，营养物质在水柱中被回收，而不是沉得足够深以发生演替。

我认为当前云提供商主导下的整合趋势正在降低开源领域的鲸落率，而这会对生态系统多样性产生无人追踪的二级效应。你可以衡量这一点：查看随时间推移的已死项目的 fork 和依赖图，统计有多少新项目引用了已死的依赖项，比较 npm、crates 和 rubygems 中鲸落的半衰期。某些生态系统的海底是否更深、分解更慢、结构性影响更持久？数据存在于包注册表和 forge API 中，但我还没看到有人问这个问题。

一个每个大型项目都被平台公司拥有的开源生态系统，无论这些项目是被无限期维护还是在死亡时被悄悄吸收，都是一个富集和化能合成阶段很少有机会发展、依赖鲸落为食的小型特殊生物永远没有机会进化的生态系统。最健康的生态系统有稳定的鲸落供应，这是个奇怪的东西去为之欢呼，因为这意味着希望大型项目死亡，除非深海海底没有其他食物来源。
