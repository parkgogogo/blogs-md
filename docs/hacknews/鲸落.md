Title: Whale Fall | 鲸落

---

When a whale dies in the open ocean, its body sinks to the abyssal seafloor and becomes an ecosystem. 

当一头鲸鱼在开阔的海洋中死去，它的尸体会沉入深渊海底，成为一个生态系统。

Marine biologists call this a *whale fall*, and the carcass sustains life through three overlapping stages: mobile scavengers strip the soft tissue within months, enrichment opportunists colonize the bones and surrounding sediment for years, and chemosynthetic bacteria feed on the skeleton itself for decades, converting the lipids stored in bone into energy that supports entire specialized communities. 

海洋生物学家将这种现象称为「鲸落」（whale fall），而尸体会在三个相互重叠的阶段中维持生命：移动清道夫在数月内撕食软组织，富集机会主义者会在数年内占据骨骼及其周围的沉积物，化能合成细菌则会在数十年间以骨骼本身为食，将储存在骨骼中的脂质转化为能量，支持整个特殊生物群落。

A single whale fall can sustain life for fifty years on an otherwise barren seafloor.

一次单独的鲸落就能在原本贫瘠的海底维持生命长达五十年。

Michael Winser mentioned whale falls to me offhand while we were talking about what happens to the dependency graph of an abandoned project, and the term has been stuck in my head since.

迈克尔·温瑟（Michael Winser）在我们讨论被遗弃项目的依赖图会发生什么时随口提到了鲸落，而这个词在我脑海中挥之不去。

---

A large open source project goes unmaintained. 

一个大型开源项目无人维护了。

Maybe the maintainer burned out, maybe the company behind it pivots. 

也许维护者精疲力竭，也许背后的公司转型了。

The project stops updating but doesn't disappear. 

项目停止更新，但并未消失。

It sits quietly on GitHub accumulating issues, the date of the last commit drifting further into the past, and then someone forks it to merge an urgent patch. 

它静静地躺在 GitHub 上积累着 issue，最后一次提交的时间越来越远，然后有人 fork 了它来合并最紧急的补丁。

If the project was popular enough, multiple forks appear, competing for users like hagfish competing for blubber, though most forks die quickly and only one or two survive on the strength of someone having enough spare time or institutional support to keep them going. 

如果项目足够受欢迎，会出现多个 fork，像盲鳗争夺鲸脂一样竞争用户，不过大多数 fork 很快就消亡了，只有一两个凭借某个人有足够的时间或机构支持来维持而存活下来。

OpenOffice became LibreOffice this way, MySQL became MariaDB, Hudson became Jenkins—each a scavenger fork that grew into the legitimate replacement through a familiar sequence: the fork announcement, the migration guide, and the blog post about why you should switch.

OpenOffice 就是这样变成了 LibreOffice，MySQL 变成了 MariaDB，Hudson 变成了 Jenkins——每一个都是清道夫式的 fork，通过一个熟悉的流程成长为正统替代品：fork 公告、迁移指南，以及「为什么你应该切换」的博客文章。

---

Then smaller projects start extracting specific modules, or building tools for the dead project's data formats. 

随后，更小的项目开始提取特定模块，或构建针对已死项目数据格式的工具。

Google Reader wasn't open source, but the same thing happened when it shut down: Feedly, Miniflux, FreshRSS, Tiny Tiny RSS and a dozen others scrambled to fill the vacuum, several of them implementing the Google Reader API or the Fever API—not because those APIs were good, but because years of RSS clients had been built against them. 

Google Reader 不是开源的，但它关闭时也发生了同样的情况：Feedly、Miniflux、FreshRSS、Tiny Tiny RSS 等十几个项目争先恐后地填补真空，其中好几个实现了 Google Reader API 或 Fever API——不是因为这些 API 有多好，而是因为多年的 RSS 客户端都是按照这些接口构建的。

The license doesn't matter. 

许可证无关紧要。

The interface is public, other software depends on it, and that is enough.

接口是公开的，其他软件依赖于它们，这就足够了。

---

Then the structural skeleton—the protocols, file formats, and API contracts—continues to support specialized communities that might not even know where the bones came from. 

然后，结构性的骨架——协议、文件格式和 API 契约——继续支撑着特殊的社区，而这些社区甚至可能根本不知道这些骨骼从何而来。

The OpenDocument format outlived the OpenOffice community that created it, sustained by document format libraries across dozens of language ecosystems. 

OpenDocument 格式比创造它的 OpenOffice 社区活得更久，由数十个语言生态系统中的文档格式库维持着。

Docker donated its container runtime and image format to the Open Container Initiative in 2015. 

Docker 在 2015 年将其容器运行时和镜像格式捐赠给了开放容器倡议（Open Container Initiative）。

The OCI specs now define how containers work regardless of runtime. 

OCI 规范现在定义了容器的工作原理，无论运行时是什么。

Docker's own dominance faded; the specs did not. 

Docker 自身的统治地位逐渐消退；规范却没有。

Tree-sitter was built for Atom and became the syntax engine inside Zed, Neovim, Helix, and most of the editors released in the last few years after GitHub archived Atom.

Tree-sitter 是为 Atom 构建的，在 GitHub 将 Atom 归档后，它成为了 Zed、Neovim、Helix 以及过去几年发布的大多数编辑器内部的语法引擎。

## Succession | 演替

---

I've noticed a recurring pattern in unmaintained libraries: the recolonization of continuity. 

我注意到无人维护的库存在一个反复出现的模式：连续性的重新定殖。

A project goes quiet, someone forks it, other projects start depending on that fork, and then the maintainer of *that* fork burns out and the whole cycle repeats at a smaller scale. 

一个项目沉寂了，有人 fork 它，其他项目开始依赖这个 fork，然后那个 fork 的维护者也精疲力竭，整个循环在小规模上重复。

Each generation of fork is usually smaller than the last, with fewer contributors and a narrower user base, until eventually the concept itself migrates rather than the code. 

每一代 fork 通常都比前一代更小，贡献者更少，用户群更窄，直到最终这个概念本身迁移了，而不是代码。

Someone in some other language ecosystem looks at the accumulated wreckage and decides to rewrite the concept from scratch, carrying the design ideas forward but leaving the implementation behind.

某个其他语言生态系统中的人看着堆积的残骸，决定从头重写这个概念，带着设计理念前进，但将实现抛在身后。

---

Sass went through this. 

Sass 经历了这个过程。

The original reference implementation was a Ruby gem. 

最初的参考实现是一个 Ruby gem。

When Ruby's performance became a bottleneck, LibSass rewrote it in C++, with the sassc gem wrapping it for Ruby users. 

当 Ruby 的性能成为瓶颈时，LibSass 用 C++ 重写了它，sassc gem 则为 Ruby 用户包装了它。

Then LibSass itself was deprecated in favor of Dart Sass, which is now the canonical implementation. 

然后 LibSass 本身被弃用，取而代之的是 Dart Sass，现在这是正统实现。

The concept migrated from Ruby to C++ to Dart over the course of a decade, each rewrite benefiting from the accumulated bug reports and design arguments of the previous generation, and at every stage there were wrapper libraries in other ecosystems feeding on the structural skeleton of the Sass language spec. 

这个概念在十年间从 Ruby 迁移到 C++ 再到 Dart，每一次重写都受益于前代积累的 bug 报告和设计争论，而在每个阶段，都有其他生态系统中的包装库以 Sass 语言规范的结构骨架为食。

Most people writing Sass today have no idea it started as a Ruby gem.

今天大多数写 Sass 的人根本不知道它最初是一个 Ruby gem。

---

The recolonization of continuity has a bad failure mode. 

连续性重新定殖有一个糟糕的失败模式。

Edera found a differential parsing bug in Rust's tar-rs library that affected every downstream fork: tar-rs itself, async-tar, tokio-tar, and multiple internal forks maintained by companies like Astral (whose fork is built into the uv package manager). 

Edera 在 Rust 的 tar-rs 库中发现了一个差异解析 bug，影响了下游的每一个 fork：tar-rs 本身、async-tar、tokio-tar，以及 Astral 等公司维护的多个内部 fork（他们的 fork 内置于 uv 包管理器中）。

Coordinating disclosure meant reaching out to about twenty entities in a fragmented fork graph, three of four library versions unmaintained and several maintainers unreachable. 

协调披露意味着要联系一个碎片化的 fork 图谱中约二十个实体，其中四个库版本中有三个无人维护，还有几个维护者无法联系。

The vulnerability existed because code was copy-forwarded through successive forks without anyone re-auditing the PAX header parsing they had all inherited from the original. 

这个漏洞存在的原因是代码通过连续的 fork 被复制转发，而没有人重新审计它们都从原始版本继承的 PAX 头解析。

The bug was in the bones, inherited by every fork. 

这个 bug 一直存在于骨骼中，被每一个 fork 继承。

Figuring out which forks of a package are affected by a security advisory is a problem I'm researching separately, because nobody currently has good tools for it.

发现某个包的哪些 fork 受到安全公告影响是一个我正在单独研究的问题，因为目前没有人有好的工具来解决它。

---

CentOS after the Stream pivot is the same pattern at operating system scale: Rocky Linux and AlmaLinux split as forks, smaller RHEL-compatible rebuilds appear in the enrichment layer around them, and the underlying structural skeleton—RPM packaging, systemd conventions, filesystem hierarchy—remains constant regardless of which specific distro is alive or dead at any given moment.

CentOS 在转向 Stream 之后，是同样模式在操作系统规模上的体现：Rocky Linux 和 AlmaLinux 分出了 fork，更小的 RHEL 兼容重建版本在它们周围的富集层出现，而下层的结构性骨架——RPM 打包、systemd 惯例、文件系统层级——无论哪个特定发行版在任何给定时刻是死是活，都保持不变。

## License Changes | 许可证变更

---

When a project switches from an open source license to a source-available license, the scavenger stage triggers almost immediately, often starting before the change even takes effect. 

当一个项目从开源许可证切换到源代码可用许可证时，清道夫阶段几乎立即触发，往往在变更生效之前就开始了。

Redis to Valkey, Elasticsearch to OpenSearch, Terraform to OpenTofu—the pattern is now familiar enough that the community has ritualized it: rush to fork the last open commit, brief competition, then consolidation around one or two survivors. 

Redis 到 Valkey，Elasticsearch 到 OpenSearch，Terraform 到 OpenTofu——这个模式现在已经足够熟悉，社区已经将其简化为一种惯例：rush 去 fork 最后一个开放的提交，短暂竞争，然后整合到一两个幸存者周围。

In these cases the organism isn't fully dead. 

这些情况下，生物体并没有完全死亡。

Redis the product still has revenue and a roadmap. 

Redis 这个产品仍然有收入和路线图。

But from the perspective of the open source ecosystem, the body of code has stopped accepting external contributions, and the forks begin to diverge from the original the way MariaDB diverged from MySQL.

但从开源生态系统的角度来看，代码主体已经停止接受外部贡献，fork 开始偏离原始版本，就像 MariaDB 偏离 MySQL 那样。

---

The abstraction layer is what persists. 

抽象层是持续存在的部分。

Every project that integrated the open version faces a choice between following the proprietary version or switching to the fork, and many simply build a compatibility shim. 

每个集成了开放版本的项目都面临一个选择：是跟随专有版本还是切换到 fork，而很多人干脆构建一个兼容垫层。

These shims often outlast the controversy that spawned them, still feeding on the skeleton of the original API years after the license debate has cooled.

这些垫层往往比引发它们的争议活得更久，在许可证辩论冷却多年后，仍然以原始 API 的骨架为食。

## Sun Microsystems

---

Oracle's acquisition of Sun in 2010 was less a single whale fall than a pod of whales dying at sea simultaneously. 

甲骨文在 2010 年收购 Sun，与其说是单个鲸落，不如说是一群鲸鱼同时在海上死去。

Java, Solaris, ZFS, DTrace, VirtualBox, NetBeans, GlassFish, Hudson, MySQL—each sank to a different seafloor and triggered its own succession. 
Java、Solaris、ZFS、DTrace、VirtualBox、NetBeans、GlassFish、Hudson、MySQL，每一个都沉到了不同的海底，各自引发了演替。

Some produced single dominant forks (Hudson to Jenkins, ZFS to OpenZFS), some fragmented into competing lineages (MySQL alone fed MariaDB, Percona, and briefly Drizzle, which itself became a smaller whale fall after it was abandoned), and some wandered between foundations before stabilizing (NetBeans to Apache, GlassFish to Payara and the broader Jakarta EE ecosystem). 

有的产生了单一的统治性 fork（Hudson 到 Jenkins，ZFS 到 OpenZFS），有的分散成相互竞争的谱系（仅 MySQL 就滋养了 MariaDB、Percona，以及短暂的 Drizzle，而 Drizzle 本身在被遗弃后成为了一个更小的鲸落），还有的在基金会之间辗转后才稳定下来（NetBeans 到 Apache，GlassFish 到 Payara 以及更广泛的 Jakarta EE 生态系统）。

The underlying structural skeleton—JVM bytecode formats, ZFS disk formats, MySQL wire protocol—still carries load in projects whose developers never heard of Sun.

下层的结构性骨架——JVM 字节码格式、ZFS 磁盘格式、MySQL 线协议——在开发者从未听说过 Sun 的项目中仍然承担着负载。

## Shallow Water | 浅水区

---

Some projects die in shallow water, where the body gets recycled quickly. 

有些项目在浅水区死去，尸体会被快速回收。

Acqui-hires work this way: the company is absorbed, the code becomes proprietary or is archived, and the nutrients are dispersed with the people rather than accumulating in a public carcass that others can feed on. 

收购雇佣（Acqui-hire）就是这样运作的：公司被吞并，代码变成专有或被归档，知识随着人员分散而不是积累在其他人可以取食的公共尸体中。

Corporate consolidation has a similar effect, because when a large independent project gets folded into a platform company's proprietary service, the nutrients are recycled in the water column rather than sinking deep enough for succession to occur.

企业整合也有类似效果，因为当一个大型独立项目被折叠进平台公司的专有服务时，营养物质在水柱中被回收，而不是沉得足够深以发生演替。

---

I think the current consolidation trend under cloud providers is reducing the whale fall rate in open source, and this has secondary effects on ecosystem diversity that nobody is tracking. 

我认为当前云提供商主导下的整合趋势正在降低开源领域的鲸落率，而这会对生态系统多样性产生无人追踪的二级效应。

You could measure this: look at forks and dependency graphs of dead projects over time, count how many new projects depend on dead dependencies, compare half-lives of whale falls in npm versus crates versus rubygems. 

你可以衡量这一点：查看随时间推移的已死项目的 fork 和依赖图，统计有多少新项目引用了已死的依赖项，比较 npm、crates 和 rubygems 中鲸落的半衰期。

Do some ecosystems have deeper seafloors, slower decomposition, more persistent structural effects? 

某些生态系统的海底是否更深、分解更慢、结构性影响更持久？

The data exists in package registries and forge APIs, but I haven't seen anyone ask the question yet.

数据存在于包注册表和 forge API 中，但我还没看到有人问这个问题。

---

An open source ecosystem where every large project is owned by a platform company, whether those projects are maintained indefinitely or quietly absorbed at death, is an ecosystem where the enrichment and chemosynthesis stages rarely get to develop and the small specialized organisms that depend on whale falls never get a chance to evolve. 

一个每个大型项目都被平台公司拥有的开源生态系统，无论这些项目是被无限期维护还是在死亡时被悄悄吸收，都是一个富集和化能合成阶段很少有机会发展、依赖鲸落为食的小型特殊生物永远没有机会进化的生态系统。

The healthiest ecosystems have a steady supply of whale falls, which is a strange thing to cheer for, because it means hoping large projects die—unless the deep seafloor is where all the food is.

最健康的生态系统有稳定的鲸落供应，这是个奇怪的东西去为之欢呼，因为这意味着希望大型项目死亡，除非深海海底没有其他食物来源。

## Critical Thinking Commentary | 批判性思考评论

### 作者核心论点分析

本文作者以「鲸落」这一精妙的海洋生物学隐喻为核心，构建了一个关于开源软件生命周期的宏大叙事。作者的核心论点是：**开源项目的「死亡」并非终结，而是一个生态演替的开始**——就像鲸鱼尸体滋养深海生态系统一样，被遗弃的开源项目会通过 fork、协议继承、格式兼容等方式，持续为整个软件生态提供养分。

文章的结构层层递进：从单个项目的清道夫式 fork（LibreOffice、MariaDB），到接口协议的持久化（OCI、OpenDocument），再到跨代际的概念迁移（Sass 从 Ruby 到 Dart），最后上升到对云时代开源生态的深刻忧虑。这种由微观到宏观的论述方式极具说服力。

### 优点与亮点

1. **隐喻的贴切性**：「鲸落」这个比喻选得极为精准。它不仅形象地描述了大型项目「死亡」后资源被逐步分解利用的过程，更暗示了这种「死亡」对生态系统的重要性——没有鲸落，深海将是贫瘠的。

2. **历史案例的丰富性**：作者对开源历史了如指掌，从 Sun Microsystems 的解体到最近的 Redis/Elasticsearch 许可证变更，案例覆盖广泛且有深度。特别是 Sass 的三代迁移（Ruby→C++→Dart）展现了概念延续与代码弃用的辩证关系。

3. **安全视角的警示**：关于 tar-rs 漏洞在 fork 图谱中扩散的讨论，揭示了「连续性重新定殖」模式的黑暗面——bug 也会随代码一同继承，而碎片化的 fork 生态使安全协调变得异常困难。

4. **云时代的批判性洞察**：文章结尾对云厂商整合趋势的忧虑发人深省。作者敏锐地指出，当大型项目被平台公司「吸收」而非「开放死亡」时，整个开源生态会失去演替和创新的机会。

### 局限性与可商榷之处

1. **隐喻的边界问题**：虽然「鲸落」比喻很精彩，但软件与生物有机体毕竟不同。软件的「死亡」是可逆的（维护者回归），而鲸鱼的死亡是绝对的。作者对此区分不够明确。

2. **对「成功 fork」的选择偏差**：文章列举的 LibreOffice、MariaDB、Jenkins 都是成功的幸存者，但大量失败的 fork 被一笔带过。清道夫阶段的「竞争激烈、大多数死亡」本应得到更多关注。

3. **云厂商角色的复杂性**：作者将云厂商描绘为开源生态的威胁，但忽略了它们同时也是重要贡献者（如 Google 的 Kubernetes、Microsoft 的 VS Code）。项目的「被吸收」未必总是坏事，有时意味着更稳定的长期维护。

4. **缺乏实证数据支撑**：作者承认「数据存在于包注册表中」，但文章本身依赖的是案例分析和轶事证据。如果能有一些量化分析（如平均 fork 存活时间、依赖死项目的活跃项目比例），论证会更有力。

### 我的批判性视角

我认为作者的核心洞察——**开源的价值不仅在于活跃项目，也在于「死亡」项目的遗产**——是完全正确的，但需要补充几个维度：

**第一，「鲸落」的养分质量有差异**。并非所有死项目都同样有价值。一个设计良好、文档完备、架构清晰的项目（如 Docker）能成为优质的「骨架」；而一个技术债务缠身、架构混乱的项目，其「尸体」可能更多是污染而非养分。作者忽略了项目「生前」质量对「死后」生态价值的决定性影响。

**第二，社区韧性比「死亡」本身更重要**。健康的开源生态不仅需要「鲸落」，更需要有能力消化这些养分的「分解者社区」。为什么有些生态（如 Python、Node.js）能产生丰富的 fork 和替代方案，而有些则不能？这涉及到社区文化、治理模式、工具链成熟度等深层因素。

**第三，云厂商的「吸收」与「鲸落」并非完全对立**。事实上，云厂商的参与有时能延缓项目的死亡（提供更多资源），有时能加速概念的迁移（通过开源核心+闭源服务的模式）。真正的危险不是「吸收」，而是**锁定**——当云厂商通过 API 设计、数据格式或网络效应形成事实垄断时，才是真正的生态危机。

### 启示与延伸思考

这篇文章的深层启示在于：**开源软件的生命周期管理需要被纳入战略考量**。对于项目维护者，应考虑「身后事」——如果必须放弃项目，如何设计架构和协议使其成为优质的「鲸落」而非污染源？对于依赖方，应关注「骨架」的健壮性——依赖于协议/格式而非特定实现，构建兼容层而非硬依赖。

更进一步，整个开源社区需要建立更好的「生态监测」机制：追踪项目的「健康指标」（维护活跃度、响应速度、社区参与度），在「病危」阶段提前介入（寻找继任维护者、转移至基金会），在「死亡」阶段有序处理（归档、标记、迁移指南），使「鲸落」成为可预期的、可控的生态事件，而非突如其来的灾难。

最后，关于云厂商角色的讨论提醒我们：**开源生态的健康不仅需要「自由」，也需要「多样性」**——多样化的资金来源、多样化的治理模式、多样化的技术栈。当任何一个单一力量（无论是云厂商还是其他）占据主导地位时，「鲸落」的供给就会减少，深海终将贫瘠。
