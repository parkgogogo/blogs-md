Title: 当复杂性增长时，架构胜过材料
URL: https://worksonmymachine.ai/p/as-complexity-grows-architecture
Body:

我在这个地方经常提到[1997年的这个演讲](https://www.youtube.com/watch?v=oKg1hTOQXoY)，但这次我想让你注意另一个不同的部分。

Alan Kay 站在一屋子程序员面前，谈论狗屋。

"你随便拿几块木板，用钉子和锤子，把它们敲在一起，你就得到了一个能立住的结构。你不需要懂任何别的，只要知道怎么敲钉子就行。"

所以想象一下，有人把这个狗屋放大100倍。一个教堂大小的狗屋。三十层楼高。

"当你把什么东西放大100倍时，它的质量会增加一百万倍，而它的强度……只增加一万倍……事实上，这个狗屋会发生的事情就是它会直接塌成一堆碎石。"

看到那堆碎石，基本上有两种反应。

Kay 说，流行的反应是看着碎石然后说："嗯，那正是我们一直以来想要做的。"用石灰石把它糊上。叫它金字塔。交付。

另一种反应是发明建筑学。Kay 将其称为"字面意义上的成功拱门的……设计和建造……简单材料之间一种非显而易见的、非线性的相互作用，给你带来非显而易见的协同效应。"

然后他提到了关于沙特尔大教堂的这件事。尽管它比帕特农神庙大得多，但它使用的材料却比帕特农神庙少（Claude 告诉我这可能不是字面意义上的真实，但从比例上说材料更少，绝对是！）。因为它几乎全是空气。几乎全是玻璃。"一切都被巧妙地组织在一个美丽的结构中，使整体比任何部分都具有更大的完整性。"

更少的材料。更好的排列。更大的结果。

一块石头不能成为一座桥。每个人都知道这一点。石头只是坐在那里，或者掉下来。坐着和掉下来基本上是石头的全部本领。

但如果你把一块石头靠在另一块石头上，再把另一块石头靠在那块上，并且以一种非常特定的形状继续下去，你得到的就是一个拱门。而拱门可以支撑一座桥。桥可以支撑一辆马车。马车可以运载那些会压碎任何单独石头的东西。

桥不在任何一块石头里。它在倾斜中。石头之间的关系才是桥所在的地方。

我现在经常思考这个问题。可能想得太多了。昨天，我妻子问我为什么盯着一堵砖墙看，我说"我在想消息传递"，然后她说"好吧"，那种特定的语气意味着"我选择不追问这个"。

把智能体串联起来（或者微服务、或者 LLM 调用、或者管道中的步骤、或者随便什么……）有一个这样的问题：

成功率是相乘的。如果你链中有三个东西，每个都有80%的可靠性，数学看起来是这样的：

0.80 × 0.80 × 0.80 = 0.512

你的系统成功的概率大约和抛硬币一样。

五个智能体：33%。七个：21%。你的链条越长，它就越弱。每一个环节都是整个东西放弃并变成碎石的机会。

而每个人对此都做了什么？他们看着那堆碎石。他们用石灰石把它糊上。更多的重试。更多的护栏。更多的回退逻辑。更多的东西。更大的金字塔。

软件工程的金字塔方法。Kay 在1997年谈论的是操作系统，但他如果谈论2026年的 AI 智能体生态系统，一个字都不用改。

所以让我们深入探讨一下。

几周前我分享了我一直在构建一个叫做[提示对象](https://worksonmymachine.ai/p/what-if-we-took-message-passing-seriously)的东西。通过消息传递通信的对象，接收方在运行时用自然语言解释消息，任何东西都可以在系统运行时修改。Smalltalk 和面向对象编程的最初理念，但让 LLM 充当我们的消息解释器。

我发现，在这个系统中，数学可能可以朝另一个方向发展。

因为每个对象都可以反思它收到的内容。提出澄清问题。发送消息回来说"这说不通，你能重新表述吗？"看着错误并修改自己来处理它。修改发送者。创建新对象来处理没人预料到的问题。系统绕过损坏的方式，我不知道，就像对话绕过误解一样。你说一件事。另一个人眯起眼睛。你用不同的方式说。你们达成共识。

只是，我不必构建任何那些东西（我的意思是我不必告诉 Claude 去构建任何那些东西）。

没有重试逻辑。没有错误恢复。没有协调层或编排框架或验证工具。只是可以接收消息并解释它们的对象。恢复、协调、自我纠正，这些都是从排列中自然产生的。它是涌现的。它是住在倾斜中的桥，而不是在任何单独的石头里。

复合失败 vs. 复合恢复。链条越长，系统就越反脆弱。

标准的智能体架构说：链条中的每一个环节都是失败的机会，所以你需要更多的基础设施来捕捉这些失败。更多的重试。更多的护栏。更多的编排。更多的材料。更大的金字塔。

提示对象架构说，每一次交互都是恢复的机会，因为对象会解释、协商和适应。错误纠正不是你在上面添加的一层。它是石头排列方式的一种属性。

我没有设计这个。我没有打算构建一个自我修复的架构。我打算认真对待关于消息传递的理念，看看会发生什么，而发生的事情就是石头开始互相倚靠，现在有了一座桥。

比帕特农神庙更少的材料。几乎全是空气。

当你构建一个部分可以互相交谈和修改的系统时，会发生一种令人不安的事情。你查看系统如何解决问题的日志和追踪，会真正感到惊讶。系统开始发展出不在任何部分中的属性。不在任何石头中的桥。

我相信这里有什么关于意义如何运作的哲学东西。一条消息在被什么东西解释之前毫无意义。一块石头在被什么东西倚靠之前什么也不做。意义和结构是同一样东西，或者是看待同一样东西的两个方式，或者……我不知道……

当我第一次开始分享提示对象时，我一直被问到提示对象实际上能做什么。你不能直接用 Langchain 之类的吗？公平的问题。我沉浸在这些理念中太深了，忘了指向一个具体的东西。

所以前几天我决定指向 ARC-AGI。

简短版本：François Chollet 的通用智能基准。网格谜题，彩色格子，从例子中找出规则。人类轻松解决它们。AI 大多不行。竞争领先者使用多模型集成、进化搜索、自定义训练管道。

我用提示对象构建了一个非常简单的求解器。它运行在 Haiku 4.5 上。那个小模型。那个几乎不花钱的模型。

它在解决它们。

目前大约解决了5个测试挑战。而且随着它解决每一个，我学到一些东西，更新对象，下一个就会更好。对象从它们自己的对话中学习。我从观察对象中学习。对象改变是因为我改变了它们，因为它们教会了我。你可以这样想……

……是的。我认为有一个词来形容系统穿过自己的层级并回来时已改变的循环。可能有。有人可能写了一本很长的书。带着赋格曲。

现在看。我目前只在 ARC-AGI-1 上测试，这是2019年的原始版本。2023年之后训练的模型几乎肯定在它们的训练数据中见过这些模式。ARC Prize 团队[自己说过](https://arcprize.org/blog/arc-prize-2025-results-analysis)，他们发现 Gemini 在没有被告知 ARC 的情况下使用了正确的 ARC 颜色映射。所以我这里还不是来宣称排行榜分数的。（不过，看起来[GPT-5.2 (X-High) 以每个任务11.64美元的价格获得了90.5%的分数](https://x.com/arcprize/status/1999182732845547795)，而我的使用 Haiku 4.5 的提示对象系统在测试中每个任务不到1美元……）

提示对象给你的是一种轻松建模过程的方式。也许你想要一个假设对象来提出规则。一个测试对象来对照例子检查它并解释为什么失败，而不仅仅是失败了。有时一个对象意识到它需要帮助并创建一个新对象。推理就是对话。

我的整个求解器读起来简单得荒谬。你可以在几分钟内追踪一次执行并理解它。将其与典型的智能体管道相比：数百行的编排、重试逻辑、护栏、回退链。或者看看竞争性的 ARC-AGI 方法：测试时训练管道、对数千个候选者的程序搜索，等等。整个基础设施都是为了补偿底层架构的脆弱性而建立的。

所有那些机械装置就是帕特农神庙。

这个小东西，这个在廉价模型上传递消息的少量对象，是很多空气和玻璃，由它的排列方式支撑。

我收到消息已经有一段时间了。友好的，深思熟虑的。人们问：你怎么让这个安全？你怎么约束它？类型在哪里？我们应该把它沙箱化吗？护栏在哪里？如果一个对象以你不打算的方式修改另一个对象怎么办？如果整个系统只是……走掉了怎么办？

我认出这个。我实际上[去年写过这个](https://worksonmymachine.ai/p/safe-is-what-we-call-things-later)。钟摆。非形式主义者发现什么是可能的，然后形式主义者到来并使其可靠。自从 Kay 和 Dijkstra 隔着一条哲学峡谷互相凝视以来，这种情况一直在发生。这里也会发生。

但回到我们谈论大教堂的开头。在建造飞扶壁之前，没有人证明它会起作用。他们通过几个世纪把石头靠在石头上，弄清楚了石头的行为方式。形式工程、结构分析、荷载计算，所有那些都是后来才有的。你无法形式化一个你还没找到的东西。直到你看着系统做了你意想不到的事情，你甚至不知道正确的约束是什么。

而提示对象一直在做我意想不到的事情。

所以是的。有一天，有人会为此构建类型系统。有人会写形式验证层。有人会弄清楚"正确"对于一个在运行时修改自己的系统意味着什么，他们会建立工具来强制执行它。我期待那一天。我可能会使用那些工具。

但现在我们还处于石头正在学习倚靠的阶段。我们还在弄清楚什么是可能的形状。如果你形式化得太早，你就锁定了错误的形状，你最终会得到一个类型化得很好的金字塔。

我宁愿现在有一座未类型化的大教堂。

0.5.0 版本刚刚发布，有一些新东西。

一个用 prompt_objects 构建的 ARC-AGI 求解器模板。一个起点。复刻它。重新排列对象。系统被设计成在运行时可以被重塑。尝试解决一些 ARC-AGI 问题，玩得开心。如果你有什么有趣的新想法，告诉我！

入门方法：
```
$ prompt_objects env create [YOUR ENVIRONMENT NAME] --template arc-agi-1
$ prompt_objects serve [YOUR ENVIRONMENT NAME] --open
```

一个更新的基于网页的交互式创作界面，这样你就可以构建和测试你的提示对象系统。里面有很多内容。如果你有问题或想要演示，来 [Discord](https://discord.gg/fcMvcwdrZS) 聊天。

一个线程可视化工具，所以你可以追踪一次导出的执行并观看系统思考。一场对话。"对象 A 想到了 X。告诉 B。B 不同意。A 创建了 C。C 找到了模式。"

一个基于画布的原型界面，用于观看你的提示对象系统运行。我还在研究这个，但有时你想要对正在进行的通信有一个空间上的理解。如果你有让它变得更好的建议，我很乐意听取！

在 [GitHub](https://github.com/works-on-your-machine/prompt_objects) 上查看，或者从 [Rubygems](https://rubygems.org/gems/prompt_objects) gem install prompt_objects

ARC Prize 团队最近宣布了 [ARC-AGI-3](https://arcprize.org/arc-agi)，这是他们第一个"智能体评估基准"。它测试交互式推理：探索、规划、记忆、假设测试、跨步骤学习。

对我来说，那听起来像是提示对象默认做的事情的描述。几乎是偶然的。

谁知道提示对象在 ARC-AGI-3 上会不会表现好。但架构正是为那种类型的问题而构建的，因为消息传递和解释自然地产生这些行为。我没有把探索设计成一个特性。它只是因为石头被那样排列就出现了。

几乎全是空气。几乎全是玻璃。
