# Ladybird adopts Rust, with help from AI

**Source:** [ladybird.org](https://ladybird.org/posts/adopting-rust/)  
**Author:** Andreas Kling (Founder & President, Ladybird)  
**Published:** Mon, 23 Feb 2026  
**HN ID:** 2626

## Summary / 摘要

Ladybird Browser Initiative announces its adoption of Rust as a successor to C++ for developing their independent web browser. After evaluating Swift and rejecting Rust initially due to OOP mismatch, they've now decided to proceed with Rust for its memory safety guarantees and mature ecosystem. Using AI coding agents (Claude Code and Codex), they successfully ported 25,000 lines of JavaScript engine code (LibJS) in just two weeks—work that would have taken months manually—with zero regressions across 65,000+ tests.

Ladybird 浏览器项目宣布采用 Rust 作为 C++ 的继任语言来开发其独立浏览器。在评估 Swift 并因面向对象不匹配而最初放弃 Rust 后，他们现在决定采用 Rust，因为它具有内存安全保障和成熟的生态系统。借助 AI 编程助手（Claude Code 和 Codex），他们在短短两周内成功移植了 25,000 行 JavaScript 引擎代码（LibJS）——这项手工完成需要数月的工作——且在 65,000 多个测试中零回归。

## Main Content / 正文

We've been searching for a memory-safe programming language to replace C++ in Ladybird for a while now. We previously explored Swift, but the C++ interop never quite got there, and platform support outside the Apple ecosystem was limited. Rust is a different story. The ecosystem is far more mature for systems programming, and many of our contributors already know the language. Going forward, we are rewriting parts of Ladybird in Rust.

我们寻找一种内存安全的编程语言来替代 Ladybird 中的 C++ 已经有一段时间了。我们之前探索过 Swift，但 C++ 的互操作性始终没有达到预期，而且在苹果生态系统之外的平台支持也有限。Rust 则是另一回事。它的生态系统在系统编程方面更加成熟，而且我们的许多贡献者已经熟悉这门语言。展望未来，我们将用 Rust 重写 Ladybird 的部分代码。

### Why Rust? / 为什么选择 Rust？

When we originally evaluated Rust back in 2024, we rejected it because it's not great at C++ style OOP. The web platform object model inherits a lot of 1990s OOP flavor, with garbage collection, deep inheritance hierarchies, and so on. Rust's ownership model is not a natural fit for that.

当我们在 2024 年初次评估 Rust 时，我们拒绝了它，因为它不太擅长 C++ 风格的面向对象编程。Web 平台对象模型继承了很多 1990 年代的 OOP 风格，包括垃圾回收、深层继承层次结构等等。Rust 的所有权模型对此并不天然契合。

But after another year of treading water, it's time to make the pragmatic choice. Rust has the ecosystem and the safety guarantees we need. Both Firefox and Chromium have already begun introducing Rust into their codebases, and we think it's the right choice for Ladybird too.

但在又一年的停滞之后，是时候做出务实的选择了。Rust 拥有我们需要的生态系统和安全保障。Firefox 和 Chromium 都已经在向代码库中引入 Rust，我们认为这对 Ladybird 来说也是正确的选择。

### Porting LibJS / 移植 LibJS

Our first target was **LibJS**, Ladybird's JavaScript engine. The lexer, parser, AST, and bytecode generator are relatively self-contained and have extensive test coverage through [test262](https://github.com/tc39/test262), which made them a natural starting point.

我们的第一个目标是 **LibJS**，即 Ladybird 的 JavaScript 引擎。词法分析器（lexer）、解析器（parser）、抽象语法树（AST）和字节码生成器相对独立，并且通过 [test262](https://github.com/tc39/test262) 拥有广泛的测试覆盖，这使它们成为自然的起点。

I used [Claude Code](https://docs.anthropic.com/en/docs/claude-code) and [Codex](https://openai.com/codex/) for the translation. This was human-directed, not autonomous code generation. I decided what to port, in what order, and what the Rust code should look like. It was hundreds of small prompts, steering the agents where things needed to go. After the initial translation, I ran multiple passes of adversarial review, asking different models to analyze the code for mistakes and bad patterns.

我使用 [Claude Code](https://docs.anthropic.com/en/docs/claude-code) 和 [Codex](https://openai.com/codex/) 进行翻译。这是由人类主导的，而非自主代码生成。我决定要移植什么、按什么顺序移植，以及 Rust 代码应该是什么样子。这是数百个小提示（prompts），在需要的地方引导 AI 助手。初步翻译完成后，我进行了多轮对抗性审查，要求不同的模型分析代码中的错误和不良模式。

### Results / 成果

The requirement from the start was byte-for-byte identical output from both pipelines. The result was about 25,000 lines of Rust, and the entire port took about two weeks. The same work would have taken me multiple months to do by hand. We've verified that every AST produced by the Rust parser is identical to the C++ one, and all bytecode generated by the Rust compiler is identical to the C++ compiler's output. Zero regressions across the board:

从一开始的要求就是两个流水线（pipelines）的输出必须逐字节相同。结果是约 25,000 行 Rust 代码，整个移植工作耗时约两周。同样的工作如果手工完成需要数个月。我们已经验证，Rust 解析器生成的每个 AST 都与 C++ 版本相同，Rust 编译器生成的所有字节码也与 C++ 编译器的输出相同。完全没有回归：

| Test Suite | Tests | Regressions |
|------------|-------|-------------|
| test262 | 52,898 | 0 |
| Ladybird regression tests | 12,461 | 0 |

| 测试套件 | 测试数量 | 回归数量 |
|----------|---------|---------|
| test262 | 52,898 | 0 |
| Ladybird 回归测试 | 12,461 | 0 |

No performance regressions on any of the JS benchmarks we track either.

我们追踪的所有 JS 基准测试也没有任何性能回归。

Beyond the test suites, I've done extensive testing by browsing the web in a lockstep mode where both the C++ and Rust pipelines run simultaneously, verifying that output is identical for every piece of JavaScript that flows through them.

除了测试套件之外，我还进行了广泛的测试，通过"锁步模式"（lockstep mode）浏览网页，在此模式下 C++ 和 Rust 流水线同时运行，验证流经它们的每一段 JavaScript 的输出都是相同的。

If you look at [the code](https://github.com/LadybirdBrowser/ladybird/pull/8104), you'll notice it has a strong "translated from C++" vibe. That's because it *is* translated from C++. The top priority for this first pass is compatibility with our C++ pipeline. The Rust code intentionally mimics things like the C++ register allocation patterns so that the two compilers produce identical bytecode. Correctness is a close second. We know the result isn't idiomatic Rust, and there's a lot that can be simplified once we're comfortable retiring the C++ pipeline. That cleanup will come in time.

如果你查看[这些代码](https://github.com/LadybirdBrowser/ladybird/pull/8104)，你会发现它带有强烈的"从 C++ 翻译而来"的风格。这是因为它*确实*是从 C++ 翻译而来的。第一次移植的首要任务是与我们的 C++ 流水线兼容。Rust 代码有意模仿 C++ 的寄存器分配模式等，以便两个编译器生成相同的字节码。正确性是次要优先事项。我们知道结果不是地道（idiomatic）的 Rust，一旦我们放心地淘汰 C++ 流水线，还有很多可以简化的地方。清理工作将在适当的时候进行。

### What's next / 未来计划

This is not becoming the main focus of the project. We will continue developing the engine in C++, and porting subsystems to Rust will be a sidetrack that runs for a long time. New Rust code will coexist with existing C++ through well-defined interop boundaries.

这不会成为项目的主要焦点。我们将继续使用 C++ 开发引擎，将子系统移植到 Rust 将是一个长期进行的旁线工作。新的 Rust 代码将通过定义良好的互操作边界与现有的 C++ 代码共存。

We want to be deliberate about which parts get ported and in what order, so the porting effort is managed by the core team. Please coordinate with us before starting any porting work so nobody wastes their time on something we can't merge.

我们希望慎重考虑哪些部分被移植以及按什么顺序移植，因此移植工作由核心团队管理。在开始任何移植工作之前，请与我们协调，这样就不会有人在我们无法合并的事情上浪费时间。

I know this will be a controversial move, but I believe it's the right decision for Ladybird's future. :^)

我知道这将是一个有争议的决定，但我相信这对 Ladybird 的未来是正确的决定。 :^)
